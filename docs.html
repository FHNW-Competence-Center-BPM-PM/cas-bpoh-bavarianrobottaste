CMS

REST-Service für Produkttexte, Bildlinks und Preis-Synchronisation per ERP-ID.

Das CMS stellt Produkttexte und Bildreferenzen auf Basis der erpId bereit. Preise können vom ERP über dieselbe fachliche Schlüssel-ID in das CMS zurückgeschrieben werden. So bleiben Texte und Medien im CMS, während Preislogik weiterhin vom ERP verantwortet wird.

Authentifizierung

Wie der CRM-Service den Gastkontext erhält

Header

Authorization: Bearer <token> bleibt für Frontend-Calls möglich, ist im geschlossenen System aber nicht zwingend.

Gastkontext

Im Prozess wird zuerst GetCustomerIdByEmail mit customerEmail aufgerufen. Die fachlichen Folgeoperationen arbeiten dann mit customerId.

Scope

Frontend-Calls können weiter nur über Bearer-Token laufen. Prozess-Calls ohne Login geben nach der Auflösung ausschließlich customerId mit.

Format

Responses kommen als SOAP-Envelope zurück; Fehler werden als SOAP-Fault geliefert.

Fehlerfälle

Professionelle Fault-Struktur

Die Schnittstelle liefert bei Fehlern nicht nur faultcode und faultstring, sondern zusätzlich einen fachlichen Detailblock mit maschinenlesbarem crm:error/code.

  <soapenv:Body>

      <faultstring>Der angegebene customerEmail-Kontext wurde nicht gefunden.</faultstring>
      <detail>
        <crm:error>
          <crm:code>customer_not_found</crm:code>
          <crm:message>Der angegebene customerEmail-Kontext wurde nicht gefunden.</crm:message>
          <crm:customerEmail>gast@example.com</crm:customerEmail>
        </crm:error>
      </detail>
  </soapenv:Body>
</soapenv:Envelope>

unknown_operation

Die angeforderte SOAP-Operation ist im CRM-Service nicht bekannt.

Weder Bearer-Token noch customerId wurden geliefert.

Roboter G1

REST-Service zur Ausführung von Unitree-G1-Armgesten über HTTP.

g1-api stellt eine FastAPI-REST-Schnittstelle bereit, um Arm-Aktionen eines Unitree G1 auszuführen. Die API kapselt die native Binary g1_arm_action_example, ruft sie mit einer Action-ID auf und kommuniziert über CycloneDDS mit dem Roboter.

Architektur

Technischer Ablauf

Client, UI oder Voice-Trigger senden einen HTTP-Request an FastAPI. Der Service delegiert an den UnitreeArmActionAdapter, startet die Binary mit einer Action-ID und übergibt die Kommunikation an CycloneDDS.

Client FastAPI UnitreeArmActionAdapter g1_arm_action_example CycloneDDS Unitree G1

Tisch Tablets

Web-Tablet mit WebSocket-Anbindung für Menü, Status, Offers und Interaktion am Tisch.

Die Tablet-Lösung besteht aus zwei Teilen: einem statischen Web-Client und einem FastAPI-basierten Event-Server. Der Client wird über Nginx ausgeliefert, verbindet sich per WebSocket mit /ws?table=<id> und erhält seine Inhalte ausschlieÃŸlich Ã¼ber JSON-Events. Ein externer Orchestrator kann über POST /api/push Inhalte oder Statusupdates aktiv auf einzelne Tablets oder auf alle Tablets gleichzeitig schicken.

Architektur

Datenfluss

Der Browser lädt die UI vom Client auf Port 8080. Nginx proxyt /ws intern auf den Event-Server. Tablets senden Benutzeraktionen und Handshakes per WebSocket an den Event-Server; dieser validiert die Events und leitet sie mit Retries an einen externen Webhook weiter. In Gegenrichtung pusht der Orchestrator über HTTP neue Events auf konkrete Targets wie table:12 oder auf all.

POST /api/push

HTTP-Push vom Orchestrator an ein oder mehrere Tablets. Der Event-Server erwartet eine Liste von Targets und ein freies Event-Objekt, das direkt an die aktiven WebSocket-Verbindungen weitergereicht wird. 

{
  "targets": ["table:12"],
  "event": {
    "type": "WELCOME_MESSAGE",
    "payload": {
      "text": "Willkommen an Tisch 12"
    }
  }
}

Events

Tablet -> Server

  • TABLE_CONNECTED für Handshake nach onopen.
  • CART_ADD und CART_REMOVE für Warenkorbänderungen.
  • ORDER_SUBMITTED mit den aktuellen Warenkorbpositionen.
  • SERVICE_CALLED mit tableId.
  • PAYMENT_CONFIRMED mit Betrag, Currency, Items, Receipt und Timestamp.
  • OFFER_ACCEPTED und OFFER_DECLINED für Server-Promotions.

Alle eingehenden Tablet-Events werden auf dem Event-Server per Pydantic als Objekt mit type und payload validiert und anschlieÃŸend an den externen Webhook mit source: table:<id> weitergeleitet.

Befüllung

Wie die Tablets Inhalte bekommen können

  • Initiale Befüllung per MENU_DATA: Kategorien und Items werden vom Server auf das Tablet gepusht. Ohne dieses Event bleibt das Menü leer.
  • Kontextuelle Ansprache per WELCOME_MESSAGE: individuelle Tisch-, Zeit- oder Gasttexte können als reines Text-Event eingespielt werden.
  • Operative Statusbefüllung per ORDER_STATUS: der Orchestrator kann den gesamten Bestellfortschritt inkl. Prozentwert und Nachricht steuern.
  • Verkaufsimpulse per PUSH_OFFER: Offers können ein bestehendes itemId referenzieren oder ein komplettes Item direkt mitliefern und optional den Preis überschreiben.
  • Engagement per FUN_EVENT: spielerische Inhalte lassen sich als Segmentliste mit Labels, CTA und Zusatzfeldern einspielen.
  • Broadcast oder gezielt: über targets in /api/push können Inhalte an ein einzelnes Tablet wie table:12 oder an all gesendet werden.
